REST Viewer

The REST Viewer in the Tizen SDK IDE provides the following functions for Web application development with using REST APIs:

  • Visualize the WADL document
    • Show WADL components with structure
  • Management for WADL component
    • Create, delete, modify WADL components
  • Authentication
    • Setting for API Key and OAuth 2.0
  • Test and get the result
    • Ease to test and check the result
  • Auto-generating code
    • Support code snippet for sending asynchronous request


Visualize the WADL document

The things what shown on REST Viewer are the WADL(Web Application Description Language) documents.

This viewer loads them from specific location (Enable to set the location at Preferences > Tizen SDK > REST Viewer or [] on toolbar) and classifies them as 4 typed WADL components (Application, Service, Path, API). It is possible to not only import/export WADL components but also handle them.

Figure: REST Viewer



Management for WADL component

There are 4 typed WADL components from WADL component:

  • [] Service (Application): The application element forms the root of a WADL description.
  • [] Base URI (Resources): The resources element acts as a container for the resources provided by the application.
  • [] Path (Resource): A resource element describes a set of resources, each identified by URI that follows a common pattern.
  • [] API (Method): This element describes the input to and output form an HTTP protocol method that may be applied to a resource.

This REST Viewer provides way to make new WADL components. It also possible to import or export WADL file(s) - You can handle only the selected WADL component if you open the dialog to modify it. It means that it is impossible to modify its parent or children component(s).

Note
Constraint: Some attributes are required to make structure for tree viewer even if they are optional on WADL schema.
  • []New: Open dialog to add new WADL component(s).

    Figure: The dialog for WADL component



  • The WADL document has ordered structure.(Service > Base URI > Path [> Path] > API)
  • Application (Service)
    • Title (required): A short plain text description of the element being documented.
  • Base URI (Resources)
    • Base URI (required): This attribute that provides the base URI for each child resource identifier.
    • Alias(optional)
  • Path (Resource)
    • Parent Path: It needs if resource has nested path structure.
    • Path (required): This provides a relative URI template for the identifier of the resource.
    • Content Type (default): Defines the media type for the query component of the resource URI.
  • API
    • Id (required): The identifier for the method.
    • Method (default): Indicates the HTTP method used.
    • Description(optional)
  • Common Parameter: Describes the input to the method as a collection of parameters.
    • Name: Parameter name
    • Value
    • Style: Indicates the parameter style.
      • Header: Specifies a HTTP header for use in the request.
      • Query: Specifies a URI query parameter for method.
      • File
    • Type: The type of the parameter as an XML qualified name.
    • Required: Whether the parameter is required to be present or not.


And it is also possible to import (export) from (to) WADL.

  • []Import WADL: The REST Viewer imports WADL file and shows WADL models.
    • From local WADL file or url.
  • []Export WADL: This exports selected WADL model(s) on viewer as a .wadl file.


Authentication

REST Viewer helps when you test the application with authentication. (Enable to set the location at Preferences > Tizen SDK > REST Viewer > Authentication)

  • API Key
    • API key authentication with header or parameter.
  • OAuth 2.0
    • OAuth 2.0 authentication information for RESTful API.

Test and get the result

You can test selected API on REST Viewer and get the result of it. There are three entry points to do it.

  • []Run: This button on the toolbar. And it also shows running history.
  • Run from context menu on selection
  • Save and Run(on Figure: The dialog for WADL component)

And if User Variables are set, you can use them with placeholder to run/test it.

Figure: User Variables



You can create a new profile with other variables. And you can choose active profile in REST Viewer’s user variable toolbar menu as below. If you test any APIs in REST Viewer, selected profile will be used.

Figure: Profile Selection



Figure: User Variable Selection



Then you can check the response result on Console View. And it is possible to save response structure.

Figure: The response result on Console View



Auto-generating code

The REST Viewer helps to send asynchronous request with REST APIs. There are two ways to generate code to do it.

  • Content assist
    • You can activate content assist with prefix “rest.”.
  • Drag and drop
    • 1. Select API model on REST Viewer, then drag it and drop on Web SDK HTML Editor.
    • 2. Select functions at bottom of dialog then you get the code.
